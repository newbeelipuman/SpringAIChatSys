# 模拟企业知识库素材：制度、流程与智能问答场景

本文档为毕业设计演示用的模拟企业知识库素材，内容经过抽象和改写，不对应任何真实公司制度、客户数据或内部系统。素材用于测试 RAG 系统在企业制度、流程说明、业务规则和系统运维场景中的检索、引用与问答能力。

## 1. 企业知识库建设背景

中大型组织通常会沉淀大量制度文件、流程说明、操作手册和业务规则。这些资料分散在不同部门和系统中，例如行政制度、财务报销规范、采购审批要求、信息系统账号管理办法、客户服务处理规范、项目交付标准、知识库维护手册等。员工在日常工作中经常遇到“知道问题但不知道查哪份文件”的情况，传统目录检索和关键词搜索很难快速定位准确条款。

企业知识库问答系统的目标，是将分散文档统一接入、清洗、分块、向量化和检索，并通过大语言模型生成结构化回答。系统需要同时提供答案和引用来源，让用户能够追溯回答依据，减少凭空生成和误解制度的风险。

## 2. 知识库覆盖范围

模拟知识库包含以下类型的内容：

1. 行政制度：考勤、请假、办公用品、会议室、印章和档案管理。
2. 财务制度：差旅报销、费用审批、发票规范、借款和付款流程。
3. 采购制度：采购申请、供应商询价、合同审批、验收和入库。
4. 信息系统制度：账号申请、权限变更、密码策略、系统日志和数据备份。
5. 客户服务规范：工单分级、响应时限、升级规则和回访要求。
6. 项目交付规范：需求确认、开发测试、上线评审、验收材料和问题闭环。
7. 知识库运维规范：文档版本、索引重建、问答评测、反馈处理和权限控制。

每一类内容都适合拆分成多个知识片段，用户提问时系统通过 TopK 检索召回相关片段，再把片段拼接为上下文交给模型生成答案。

## 3. 文档接入流程

文档接入是知识库建设的第一步。管理员可以上传制度文件、操作手册或业务说明文本。系统接收到文件后，需要先保存原始文件，再执行文本提取、清洗、分块和索引构建。

标准流程如下：

1. 上传原始文档，记录文档名称、来源、业务分类、版本号和上传人。
2. 提取文本内容，过滤空页、页眉页脚、目录页、重复水印和无意义字符。
3. 根据标题、段落和长度进行分块，避免单个片段过长导致检索不精准。
4. 为每个文本片段生成 embedding 向量，并保存 chunkId、docId、documentName、source、content 等元数据。
5. 将向量写入向量库，用于后续语义检索。
6. 管理员可执行检索测试，确认重点条款能够被召回。

文档接入时需要保留来源信息。回答中展示引用来源，是制度问答系统区别于普通聊天机器人的重要能力。

## 4. 文本分块规则

文本分块会影响召回质量。过短的片段会丢失上下文，过长的片段会降低语义匹配精度。模拟系统采用“先按段落，再按长度”的策略。

推荐规则：

- 优先按一级标题、二级标题和自然段切分。
- 如果单段超过固定长度，则按字符长度继续切分。
- 相邻片段之间可以保留少量 overlap，避免条款被切断后上下文不完整。
- 每个 chunk 保留文档名、标题路径、来源文件和分块序号。
- 对流程类文档，应尽量保证一个审批步骤或一个规则条款落在同一个片段中。

例如“差旅报销流程”中的“提交申请、部门审批、财务审核、付款归档”四个步骤，可以分别成为可检索片段，也可以在摘要片段中保留完整流程概览。

## 5. RAG 问答主流程

RAG 问答流程包括问题理解、向量检索、上下文构造、答案生成和引用返回。

当用户提出问题后，系统先将问题转成 embedding 向量，然后在知识库中执行 TopK 检索，召回与问题最相关的文本片段。系统再把召回片段整理为上下文，并在 Prompt 中要求模型只能基于上下文回答。如果上下文不足，模型应明确说明“当前知识库无法确认”。

标准返回内容包括：

- question：用户原始问题。
- answer：基于检索上下文生成的答案。
- citations：引用来源列表，包括文档名、片段编号、来源和相似度。
- retrievedChunks：实际召回的文本片段，便于调试和答辩展示。
- elapsedMs：处理耗时。
- mode：当前模型模式，例如 local-demo 或 spring-ai-openai。

答辩演示时，可以先展示直接问模型的回答，再展示 RAG 回答。直接问模型可能缺少依据，而 RAG 回答能够列出引用片段，说明系统具备知识追溯能力。

## 6. 差旅报销制度

员工因公出差前，应在出差管理系统中提交出差申请。申请内容包括出差事由、目的地、预计时间、交通方式、预算金额和同行人员。申请经直属负责人审批通过后，员工方可安排交通和住宿。

出差结束后，员工应在十个工作日内提交报销申请。报销材料包括审批通过的出差申请、交通票据、住宿发票、行程单、付款凭证和必要的说明材料。财务人员应核对票据真实性、金额一致性和审批链完整性。

报销流程如下：

1. 员工填写报销单，上传发票和行程证明。
2. 部门负责人审核出差必要性和费用合理性。
3. 财务专员审核票据、预算和报销标准。
4. 财务负责人复核大额费用。
5. 出纳安排付款或冲销借款。
6. 系统归档报销记录和附件。

如果缺少发票、行程单或原始审批记录，财务可以退回报销申请。员工补充材料后重新提交，不需要重新发起出差申请，但需要保留退回原因和修改记录。

住宿标准按城市级别和岗位等级确定。超过标准的部分原则上不予报销，确因会议指定酒店、突发交通调整或客户安排导致超标的，应提交情况说明并经部门负责人确认。

## 7. 采购审批制度

采购申请适用于办公设备、软件服务、项目材料、外包服务和固定资产采购。申请人需要说明采购目的、预算来源、规格要求、预计金额和交付时间。

采购金额在低额度范围内时，可由部门负责人审批后直接采购；达到中等金额时，需要采购部门询价并至少保留两家供应商报价；达到高额度时，需要组织比价或评审，并提交合同审批。

采购流程如下：

1. 申请人提交采购需求。
2. 部门负责人确认业务必要性。
3. 采购专员进行供应商询价。
4. 财务或预算负责人确认预算可用。
5. 法务或合同管理员审核合同条款。
6. 采购负责人批准采购。
7. 使用部门验收交付结果。
8. 财务根据合同、发票和验收单付款。

软件服务采购应特别关注账号数量、服务期限、数据安全条款、退出机制和售后服务。涉及客户数据、员工信息或生产系统访问权限的软件采购，需要额外经过信息安全评估。

## 8. 信息系统账号管理

新员工入职时，人力资源部门应发起账号开通申请。账号范围包括企业邮箱、办公系统、知识库系统、项目管理系统和必要的业务系统。权限分配应遵循最小权限原则，即员工只获得完成岗位职责所需的最小访问范围。

账号申请流程：

1. 人力资源部门提交入职人员基础信息。
2. 直属负责人确认岗位和权限模板。
3. 信息系统管理员开通账号。
4. 员工首次登录后修改初始密码。
5. 系统记录开通时间、操作人和权限范围。

员工调岗时，原部门权限应及时回收，新部门权限重新审批。员工离职时，应在离职日期前完成账号冻结、权限回收、资料交接和设备归还。涉及核心系统管理员权限的离职人员，应立即变更共享密钥、服务账号密码和重要系统访问策略。

密码策略要求：

- 密码长度不少于八位。
- 不得使用姓名、手机号、生日等容易猜测的信息。
- 高权限账号应启用多因素认证。
- 初始密码必须在首次登录后修改。
- 发现账号异常登录时，应立即冻结账号并通知信息安全负责人。

## 9. 客户服务工单规范

客户服务工单按照影响范围和紧急程度分为四级。

一级工单表示生产系统大范围不可用或核心客户业务中断，需要立即响应。二级工单表示关键功能异常但存在临时绕行方案，需要优先处理。三级工单表示普通功能问题、配置咨询或非紧急故障。四级工单表示建议、优化请求或一般咨询。

响应要求：

- 一级工单应在十五分钟内响应，并建立处理群同步进展。
- 二级工单应在一小时内响应，并在当天给出处理计划。
- 三级工单应在一个工作日内响应。
- 四级工单应在三个工作日内回复。

工单处理闭环包括受理、定位、解决、验证、回访和归档。客服人员不得在未确认的情况下承诺具体修复时间。涉及研发修复的工单，应关联缺陷编号或任务编号，便于后续追踪。

客户反馈中包含账号、手机号、合同金额、业务数据等敏感信息时，客服人员应按脱敏规范处理，不得将原始敏感信息复制到非授权系统。

## 10. 项目交付流程

项目交付从需求确认开始。项目经理应组织客户、产品、研发和测试共同确认需求范围。需求确认材料包括需求清单、原型图、接口说明、验收标准和上线计划。

开发完成后，测试人员根据验收标准编写测试用例。测试通过后，项目进入上线评审。上线评审关注变更范围、影响系统、回滚方案、数据备份、通知对象和上线窗口。

项目验收材料通常包括：

- 项目范围说明。
- 需求完成清单。
- 测试报告。
- 上线记录。
- 用户操作手册。
- 遗留问题清单。
- 客户验收确认记录。

如果项目存在遗留问题，应明确责任人、解决期限和影响范围。遗留问题不影响核心功能时，可以进入试运行；如果影响核心流程，应延期验收或拆分上线范围。

## 11. 知识库运营规范

知识库不是一次性建设项目，需要持续维护。每一份知识文档都应有负责人、版本号、发布日期和有效状态。过期制度应标记为失效，避免被问答系统误召回。

知识库维护人员需要定期执行以下工作：

1. 检查新增制度是否已入库。
2. 对过期文档进行下线或版本替换。
3. 根据用户反馈补充高频问题。
4. 抽查问答结果是否引用正确来源。
5. 对低质量片段进行重新清洗和分块。
6. 在文档更新后重建向量索引。

如果系统返回“当前知识库无法确认”，维护人员应判断是问题超出范围、文档缺失、分块不合理还是检索参数不合适。对于高频但无法回答的问题，应补充标准问答或制度说明。

## 12. 问答质量评测

企业知识库问答系统需要通过测试集评估效果。测试集可以包含制度问答、流程问答、边界问题和无答案问题。

常见评测维度：

- 召回准确率：相关片段是否进入 TopK。
- 答案正确性：回答是否符合制度内容。
- 引用一致性：答案是否能在引用片段中找到依据。
- 拒答能力：知识库没有依据时是否避免编造。
- 响应时间：检索和生成总耗时是否可接受。
- 可读性：答案是否结构清晰、便于员工理解。

测试示例：

问题：员工出差后多久需要提交报销？
期望依据：差旅报销制度中“出差结束后十个工作日内提交报销申请”。
期望回答：员工应在出差结束后的十个工作日内提交报销申请，并上传发票、行程单和相关审批材料。

问题：软件服务采购为什么需要安全评估？
期望依据：采购制度中涉及客户数据、员工信息或生产系统访问权限的软件采购，需要信息安全评估。
期望回答：因为软件服务可能涉及敏感数据和系统访问权限，需要确认数据安全、账号管理、退出机制和售后责任。

问题：一级客户工单多久响应？
期望依据：客户服务工单规范中“一级工单应在十五分钟内响应”。
期望回答：一级工单应在十五分钟内响应，并建立处理群同步进展。

## 13. 安全和权限要求

企业知识库系统需要控制文档访问权限。并非所有员工都能查看所有制度和业务文档。财务制度、人事信息、客户合同、项目资料和技术运维文档都可能需要分级授权。

权限控制要求：

- 用户只能检索自己有权限查看的文档。
- 回答中不得泄露未授权文档内容。
- 管理员操作需要审计日志。
- 文档下载、删除、重建索引等高风险操作需要记录操作人和时间。
- 外部模型调用时，不应传入敏感原文，除非已经完成合规审批和脱敏处理。

在毕业设计演示中，可以使用模拟数据展示权限设计，不接入真实公司数据，不上传真实账号、密钥、合同、客户信息和内部制度原文。

## 14. 答辩演示建议

答辩时可以按以下路径演示：

1. 打开状态页，展示后端健康状态、模型模式和向量库模式。
2. 在写入页一键导入演示素材，说明文档会被分块、向量化和入库。
3. 在提问页输入“员工出差后多久需要提交报销”，展示 RAG 回答和引用来源。
4. 点击“仅检索”，展示 TopK 命中的片段和相似度。
5. 点击“直接问模型”，展示没有检索增强时的回答差异。
6. 说明系统通过 citations 和 retrievedChunks 实现答案追溯。
7. 说明当前本地模式使用 hash embedding 和内存向量库，外部模型配置后可切换为真实 Spring AI 调用。

推荐演示问题：

- 员工出差后多久需要提交报销？
- 采购流程中什么时候需要供应商询价？
- 软件服务采购为什么需要安全评估？
- 新员工账号开通需要经过哪些步骤？
- 一级客户工单多久响应？
- 知识库维护人员需要定期做哪些工作？
- 如果知识库没有依据，系统应该如何回答？

## 15. 总结

本素材用于模拟企业内部制度知识库。它包含多类制度、流程和规则，适合测试 RAG 系统的文档写入、语义检索、上下文拼接、答案生成、引用追溯和答辩展示能力。素材不包含真实公司名称、真实业务数据、账号密钥或源码内容，可作为毕业设计演示中的知识库文本。
